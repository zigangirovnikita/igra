# Журнал решений и ошибок (Lessons Learned)

Этот файл содержит ключевые выводы, архитектурные решения и исправленные ошибки, которые возникли из-за неверной интерпретации ТЗ или технических ограничений.

**КАЖДЫЙ АГЕНТ ОБЯЗАН ИЗУЧИТЬ ЭТОТ ФАЙЛ ПЕРЕД НАЧАЛОМ РАБОТЫ.**

---

### 1. Ошибка: Dashboard вместо Визуальной Новеллы
**Проблема:** Изначально интерфейс игры был реализован как классический dashboard/grid с кнопками действий и логом событий на одном экране. Это полностью ломало пользовательский опыт (UX) и ощущение "игры", описанное в ТЗ.
**Решение:** Был написан кастомный `SceneEngine` (`components/scenes/SceneEngine.tsx`), который перевёл игру в формат поэкранного визуального романа.
**Вывод:** Интерфейс должен всегда подаваться последовательно (экран настройки -> нарративный текст -> выбор -> результат), а не вываливать всю информацию на пользователя одновременно. Не использовать списки действий на одном экране с результатами.

### 2. Ошибка: Использование плоских форм для онбординга
**Проблема:** Вся стартовая настройка (выбор персонажа, ниши, чека) была в одной большой HTML-форме. Это выглядело как скучный опросник.
**Решение:** Реализован 13-шаговый компонент `SetupScene.tsx`, где каждый вопрос задаётся на отдельном экране с поддерживающим текстом и визуалом.
**Вывод:** Все игровые формы должны быть разбиты на логические шаги (wizard) с элементами сторителлинга.

### 3. Ошибка: Технические мечты вместо профессиональных
**Проблема:** Изначально в целях были только базовые вещи ("Ноутбук", "Отпуск"). Это не резонировало с целевой аудиторией экспертов.
**Решение:** В конфиг `game-config.v1.json` были добавлены цели, понятные экспертам ("Уйти с постоянной работы", "Нанять ассистента", "Перестать продавать вручную", "Выйти на 300 000 ₽").
**Вывод:** Тексты и опции выбора должны всегда соответствовать болям конкретной ЦА (онлайн-школы, наставники, эксперты).

### 4. Ошибка: Математика в UI компонентах
**Проблема:** Баланс и расчеты могут случайно просочиться в React компоненты.
**Решение:** Все расчеты и изменения состояния строго изолированы в пакете `packages/game-engine`. Движок детерминирован и ничего не знает про UI.
**Вывод:** Никогда не хардкодить цены, коэфициенты и правила конверсии внутри `components/`. Вся бизнес-логика должна жить в `packages/game-engine`.
